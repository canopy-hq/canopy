Architecture Approaches

How should the Rust backend and SolidJS frontend communicate?

A

Recommended: Event-Driven IPC

Rust backend owns all PTY sessions, git operations, and filesystem access. Communicates with frontend via Tauri's typed event system + commands.


┌─────────────────────────────────┐
│ SolidJS Frontend │
│ xterm.js │ sidebar │ pane mgr │
└──────────┬──────────────────────┘
│ Tauri Commands + Events
┌──────────┴──────────────────────┐
│ Rust Backend │
│ PTY mgr │ git ops │ fs watch │
│ process detect │ workspace mgr │
└─────────────────────────────────┘

Pros

  • Clean separation — Rust does heavy lifting, JS does rendering
  • Type-safe IPC via Tauri's command system
  • PTY I/O stays in Rust, only rendered bytes go to frontend
  • Easy to add new backend capabilities

Cons

  • Slightly more boilerplate for each new command
B

WebSocket Bridge

Rust backend runs a local WebSocket server. Frontend connects via WS for PTY streaming and HTTP for commands.


Pros

  • Familiar web pattern
  • Could support remote terminals later

Cons

  • Extra complexity — managing WS connections on top of Tauri IPC
  • Duplicates what Tauri already provides natively
  • More latency than direct IPC
  • Security surface (open port)
C

Hybrid: Tauri IPC + Shared Memory for PTY

Use Tauri commands for git/workspace ops, but shared memory buffer for PTY I/O to minimize copy overhead on high-throughput terminal output.


Pros

  • Maximum throughput for terminal rendering
  • Near-zero copy for PTY data

Cons

  • Significant complexity for marginal gain
  • Tauri's event system is already fast enough for terminal data
  • Shared memory in WebView is non-trivial
  • Premature optimization
