Design — Section 3: Data Flow & PTY Architecture

How terminals, workspaces, and agent detection connect

Terminal / PTY Lifecycle

User opens tab/splits pane
│
▼
Frontend ── invoke("pty_create", { cwd, shell }) ──▶ Rust
│ │
│ │ portable-pty spawns shell
│ │ returns pty_id
│ ▼
│ Rust ── emit("pty_output", { pty_id, data }) ──▶ Frontend
│ (continuous stream via Tauri events)
▼
xterm.js writes data to terminal canvas

User types in terminal
│
▼
Frontend ── invoke("pty_write", { pty_id, data }) ──▶ Rust ──▶ PTY stdin

Agent Detection Flow

Every 2s per active PTY:

Rust → sysinfo: get child processes of PTY shell pid
│
├─ match process name against known_agents (["claude", "codex", "aider", ...])
│ ├─ matched → emit agent_status_changed { running: true, agent: "claude" }
│ └─ no match → check manual_override flag
│ ├─ manual = true → keep showing indicator
│ └─ manual = false → emit agent_status_changed { running: false }
│
└─ Frontend updates sidebar badge + tab pill
